1. IPv6 下一代 IP (p28)

为什么要 IPv6？Motivation

  • 地址耗尽：IPv4 是 32 位 (≈ 43 亿)，全球早已分配完
  • 头部更简单：固定 40 字节，路由器转发更快
  • 支持流标签 (Flow Label)：为 QoS 服务标识"流"

IPv6 头部格式 (固定 40 字节)

字段 Field位数 Bits说明
Version4= 6
Traffic Class / Priority8流量优先级
Flow Label20同一"流"的所有包打同样标签，便于路由器区分
Payload Length16负载字节数（不含 40 字节头部）
Next Header8下一个头部类型（替代 IPv4 的 Protocol；用于扩展头链）
Hop Limit8跳数限制（替代 IPv4 的 TTL）
Source Address128源地址（16 字节）
Destination Address128目的地址（16 字节）

地址表示法 Address Notation

IPv6 地址 = 128 位 = 8 组 16 位十六进制，组间用冒号分隔：

2001:0db8:0000:0000:0000:ff00:0042:8329

简写规则：

  • 每组前导 0 可省：0db8 → db8，0000 → 0
  • 一段或多段连续的全 0 可用 :: 替代（只能用一次，避免歧义）

简写示例：
2001:0db8:0000:0000:0000:ff00:0042:8329
→ 2001:db8::ff00:42:8329

IPv4 vs IPv6 关键差异

项IPv4IPv6
地址长度32 bits128 bits
头部长度20–60 字节（变长）40 字节固定
校验和有（头部 checksum）无（提高路由速度）
分片路由器可分片只在源端分片
Options头部内通过 Extension Headers
地址自动配置需 DHCP支持 SLAAC（无状态自动配置）
💡 IPv4 → IPv6 过渡机制：Tunneling 隧道中文

互联网无法一夜之间全部换成 IPv6 路由器，所以引入隧道技术：把 IPv6 包当作 payload 装入 IPv4 包，穿越只支持 IPv4 的网络段，到对端再剥出 IPv6 包。"包中包"。

4G/5G 核心网、家庭路由器（如 6to4、Teredo）都广泛使用隧道。

2. Dijkstra Link-State Routing 链路状态路由算法 (p53–54)

特点： 集中式（每个节点都知道完整拓扑）+ 迭代逐步加入节点 + 单源最短路径到所有其他节点。

符号约定 Notation

  • c(x,y) = 节点 x 到 y 的直接链路代价；若不相邻，c = ∞
  • D(v) = 当前从源到 v 的最短路径估计
  • p(v) = 路径上 v 的前驱节点
  • N′ = 已确定最短路径的节点集合

算法伪码 Pseudocode

Initialization:
  N′ = {u}          // u 是源节点
  for all v:
    if v 是 u 的邻居: D(v) = c(u, v)
    else:           D(v) = ∞

Loop:
  do
    find w ∉ N′ 中 D(w) 最小的节点
    add w to N′
    for all v 邻于 w 且 ∉ N′:
      D(v) = min( D(v), D(w) + c(w, v) )
        // 比较：原估计 vs 经 w 中转
  while N′ ≠ 所有节点
例题：求源 u 到所有其他节点的最短路径

网络如图（节点 u, v, w, x, y, z）：

边权：(u,v)=2, (u,x)=1, (v,x)=3, (v,w)=5, (v,y)=3, (x,w)=3, (x,y)=1, (w,y)=2, (w,z)=5, (y,z)=2

StepN′D(v),p(v)D(w),p(w)D(x),p(x)D(y),p(y)D(z),p(z)
0{u}2,u∞1,u∞∞
1{u,x}2,u4,x—2,x∞

最终路径：u→v(2), u→x(1), u→x→y(2), u→x→w(4), u→x→y→z(4)

复杂度：O(n²)（朴素实现）或 O(n log n)（用堆）。

3. Distance Vector Algorithm 距离向量算法 (p73)

Bellman-Ford 方程

Bellman-Ford Dx(y) = minv { c(x,v) + Dv(y) }
  • Dx(y) = 从 x 到 y 的最短代价
  • min 是对 x 的所有邻居 v 取最小
  • c(x,v) = 直接代价；Dv(y) = v 自己估计到 y 的最短代价

算法特点 Characteristics

  • 分布式：每个节点只与邻居交换信息
  • 迭代：本地代价或邻居 DV 变化才计算
  • 异步：无需全局时钟
  • 每个节点定期把距离向量（自己到所有目的地的估计）发给邻居
例题：u 的邻居有 v, x, w，已知 Dv(z)=5, Dx(z)=3, Dw(z)=3，c(u,v)=2, c(u,x)=1, c(u,w)=5，求 Du(z)
计算： Du(z) = min{ c(u,v)+Dv(z), c(u,x)+Dx(z), c(u,w)+Dw(z) }
= min{ 2+5, 1+3, 5+3 } = min{ 7, 4, 8 } = 4
→ 最短下一跳 = x

Count-to-Infinity 问题（必考点 ⭐⭐）

当某条链路代价变大或链路故障时，DV 收敛极慢——"坏消息传得慢"，可能产生路由环路与计数到无穷。

💡 Count-to-Infinity 怎么发生？怎么解决？中文

场景：A — B — C，c(A,B)=1, c(B,C)=1，所以 DA(C)=2，下一跳是 B。

现假设 A-B 之间链路失效。B 检测到，把 DB(A) 设为 ∞。但 C 还以为 DC(A)=2（经 B 到 A）。

B 看到 C 说"我到 A 距离是 2"，B 就更新自己：DB(A) = c(B,C)+DC(A) = 1+2 = 3。然后告诉 C。
C 看到 B 说 3，更新自己：DC(A) = 1+3 = 4 …… 无限递增到 ∞。

两种缓解策略：

  • Split Horizon 水平分裂：C 不向 B 通告"经 B 到 A 的路径"——既然这条路是从 B 学来的，再告诉 B 没意义
  • Poison Reverse 毒性逆转：C 向 B 通告"我到 A 的距离 = ∞"，让 B 立刻知道这条路不能再用
  • 定义"无穷"：把最大跳数限定为较小值（RIP 取 16 即无穷），防止数到天荒地老

4. TCP 传输控制协议 (p97)

TCP 核心特性

Point-to-point一个发送方对一个接收方
Reliable, in-order byte stream可靠、有序的字节流（不分消息边界）
Full duplex双向同时收发
Connection-oriented面向连接（先握手）
Flow controlled有流控（receive window）
Congestion controlled有拥塞控制（cwnd）

TCP 段头部 (Segment Header)

字段位数含义
Source Port16源端口
Destination Port16目的端口
Sequence Number32本段第一个字节在字节流中的编号
Acknowledgement Number32期望对方下一个字节的编号（累积 ACK）
Header Length4头部长度（4 字节为单位）
Flags6+SYN, ACK, FIN, RST, PSH, URG, CWR, ECE
Receive Window16接收方愿意接受的字节数（流控）
Checksum16整段校验和
Urgent Pointer16紧急数据指针
Options0–40 字节MSS, SACK, Timestamp 等

三次握手 Three-way Handshake ⭐⭐

  1. 客户端 → 服务器：SYN=1, seq=x（客户端起始序号）
  2. 服务器 → 客户端：SYN=1, ACK=1, seq=y, ack=x+1（确认 x，同时自己也选个起始序号 y）
  3. 客户端 → 服务器：ACK=1, seq=x+1, ack=y+1（确认 y）

四次挥手 Connection Teardown

💡 为什么握手是 3 次而挥手是 4 次？中文

握手 3 次：SYN+ACK 可合并成 1 个段，所以一来一回一确认 = 3 次。

挥手 4 次：关闭是半双工分别关——A 不发了不代表 B 不发了。B 收到 A 的 FIN 后先 ACK 确认（"知道你不发了"），等自己也想关时再发自己的 FIN。这个 ACK 和 FIN 通常不能合并（B 可能还要传数据），所以是 4 次。

TIME_WAIT (2·MSL)：主动关闭方最后发完 ACK 后还要等 2 倍报文最大生存时间，确保 ACK 不丢、防止旧连接的延迟分组干扰新连接。

5. Congestion Control 概念 拥塞控制 (p119)

拥塞 (congestion): 网络负载超过其容量，路由器队列溢出、丢包，整体吞吐量反而下降。

策略分类

Open-loop 开环

预防 (Prevention)：在发生拥塞前就限制流量。

  • 准入控制 (admission control)
  • 流量整形 (traffic shaping)：漏桶/令牌桶

Closed-loop 闭环

响应 (Removal)：检测到拥塞再调整。

  • Backpressure：拥塞节点向上游发信号
  • Choke packet：拥塞路由器直接告诉源主机减速
  • Implicit signaling：源端从超时/丢包推断拥塞（TCP 用这个）
  • Explicit signaling：ECN（拥塞标志位）

6. TCP Congestion Control TCP 拥塞控制 (p124)

TCP 用拥塞窗口 cwnd（congestion window）控制发送速率：实际发送 = min(cwnd, rwnd)。

三大阶段

  1. Slow Start 慢启动：cwnd 初值 = 1 MSS，每收到一个 ACK，cwnd += 1 MSS（每 RTT 翻倍 → 指数增长）
  2. Congestion Avoidance 拥塞避免：当 cwnd ≥ ssthresh，每 RTT 只增加 1 MSS（线性增长 — AIMD 的 AI）
  3. Fast Retransmit / Fast Recovery：收到 3 个重复 ACK ⇒ 立刻重传，不等超时

丢包事件 (Loss Event) 反应

事件ssthreshcwnd状态
Timeout 超时= cwnd / 2= 1 MSS回到 Slow Start
3 dup ACKs= cwnd / 2= ssthresh（或 ssthresh+3）进入 Fast Recovery → Congestion Avoidance

AIMD：Additive Increase, Multiplicative Decrease

正常时：cwnd ← cwnd + 1 MSS（每 RTT，线性增）
丢包：cwnd ← cwnd / 2（指数减）

cwnd 随时间变化（典型锯齿图）

例题：cwnd=16 MSS 时超时，下一 RTT 后 cwnd = ?
超时 → ssthresh = 16/2 = 8，cwnd = 1
下一 RTT：cwnd 从 1 慢启动指数翻倍 → 2
再下一 RTT → 4 → 8（到 ssthresh）→ 转入 CA 线性增

7. UDP 用户数据报协议 (p136)

UDP 特性

  • 无连接（connectionless）：发包前不握手
  • 不可靠：可能丢包、乱序、重复，全交给应用层处理
  • 无流控、无拥塞控制：能发多快就多快
  • 头部小：8 字节
  • 支持多播/广播

UDP 头部（仅 8 字节）

字段位数含义
Source Port16源端口（可选，为 0 表示不需要回包）
Destination Port16目的端口
Length16UDP 段总长度（含头部，最小 8）
Checksum16校验和（IPv4 可选，IPv6 必须）

TCP vs UDP 对比表（必考）

特性TCPUDP
连接面向连接（三次握手）无连接
可靠性可靠（确认 + 重传）不可靠
有序性有序字节流不保证顺序
流控有（rwnd）无
拥塞控制有（cwnd, AIMD）无
头部大小20–60 字节8 字节
速度较慢（有控制开销）快
典型应用HTTP, FTP, SMTP, SSHDNS, DHCP, 视频/语音流, 在线游戏
💡 什么时候选 UDP？中文

三种典型场景：

  1. 实时性要求高 > 可靠性：直播、视频会议、VoIP、网络游戏。丢一两个包总比卡顿一秒强。
  2. 请求-响应单次小包：DNS 查询、SNMP。建立 TCP 连接的握手开销不划算。
  3. 广播/多播：TCP 是点对点，不支持广播；只能用 UDP。

Q2. 关于 IPv4 vs IPv6，下列哪项错？

  • A. IPv6 头部包含校验和字段
  • B. IPv6 地址 128 位
  • C. IPv6 头部固定 40 字节

A. IPv6 取消了头部校验和（让路由器转发更快，可靠性由上层保证）。

Q3. Dijkstra 算法的时间复杂度（朴素实现）是？

  • A. O(n)
  • B. O(n log n)
  • C. O(n²)

C. 每轮找最小 D(w) 是 O(n)，共 n 轮 → O(n²)。用最小堆优化可到 O((n+E) log n)。

Q5. TCP 三次握手中，第 2 个段（Server → Client）携带：

  • A. SYN=1, ACK=0
  • B. SYN=0, ACK=1
  • C. SYN=1, ACK=1, ack=x+1
  • D. FIN=1, ACK=1
C. 服务器同时确认客户端 SYN 并自己也发起 SYN，所以 SYN=1, ACK=1, ack=x+1, seq=y。

Q6. 一条 TCP 连接 cwnd=16 MSS 时发生超时，超时后立即的 cwnd 与 ssthresh 分别是：

  • A. cwnd=1, ssthresh=8
  • B. cwnd=8, ssthresh=16
  • C. cwnd=16, ssthresh=8
  • D. cwnd=8, ssthresh=8
A. 超时后 ssthresh = cwnd/2 = 8，cwnd 回到 1 MSS，从慢启动开始。

Q7. AIMD 的含义是：

  • A. 线性减、指数增
  • B. 线性增（+1）、乘法减（/2）
  • C. 指数增、指数减
  • D. 指数增、线性减
B. Additive Increase, Multiplicative Decrease。慢启动是指数，但拥塞避免阶段是 AIMD：每 RTT cwnd+1，丢包 cwnd/2。

Q8. 下列哪个应用通常用 UDP？

  • A. HTTP 网页浏览
  • B. DNS 查询
  • C. SSH 远程登录

B. DNS 是简单的请求-响应小包，用 UDP（端口 53）。HTTP/SSH/SMTP 都需可靠字节流，用 TCP。
